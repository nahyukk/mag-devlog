## 웹 서버, 웹 애플리케이션 서버

- 웹서버는 정적 리소스
- 웹 애플리케이션 서버(WAS)는 정적 리소시 + 애플리케이션 로직 제공

### 웹 시스템 구성 - WAS + DB + WEB

- was도 정적 리소스를 처리할 수 있지만..
- was가 너무 많은 역할을 담당하면 서버 과부하 우려
- HTML, CSS, JS 는 아주 싸다. ⇒ web server가 하면 good
- 효율적인 리소스 관리 가능
    - 정적 리소스 많이 사용하면 Web 서버 증설
    - 애플리케이션 로직을 많이 쓰면 was 증설
- was는 잘 죽음…
- 웹서버는 잘 안죽음 - was, db 죽으면 오류화면 HTML 쓰면 된다.

## 서블릿

- urlPatterns의 URL이 호출되면 서블릿 코드 실행
- HTTP 요청 정보를 사용할 수 있는 HttpServletRequest
- HTTP 요청 정보를 제공(응답 메세지) 할 수 있는 HttpServletResponse

→ HTTP 스펙을 매우 편리하게 사용

### 서블릿 컨테이너

- 서블릿을 지원하는 was
- 생명주기 관리 - 서블릿 객체 생성, 초기화, 호출, 종료
- **싱글톤으로 관리**
    - 최초 로딩 시점에 서블릿 객체 미리 만들고 재활용
    - 모든 고객 요청은 동일한 서블릿 객체 인스턴스에 접근
    - **공유 변수 사용 주의** (로그인 했는데 다른 사람 이름이 보일 수도)
    - 서블릿 컨테이너 종료시 같이 종료
    - response는 다르지만… (이거 맞나?)
- JSP(뒤에 설명)도 서블릿으로 변환되어서 사용
- ⭐ 멀티 쓰레드 처리 지원

## 동시 요청 - 멀티 쓰레드

- 트레픽 많을 때.. 어쩌지?

### 쓰레드

- 코드를 하나하나 순차적으로 실행
- main이라는 쓰레드가 실행
- 쓰레드는 한번에 하나의 코드 라인만 수행 (없으면 자바 애플리케이션 실행 불가)
- 동시 처리가 필요하면 쓰레드를 추가로 생성

**요청마다 쓰레드를 생성**

- 장점
    - 동시 요청을 처리
    - 리소스(CPU, 메모리)가 허용될 떄까지 처리 가능
    - 하나의 쓰레드가 지연되도, 나머지 쓰레드 정상 동작
- 단점
    - 쓰레드 생성 비용이 비쌈
        - 고객 요청이 올 때마다 생성하면, 응답 속도가 늦어짐
        - 컨텍스트 스위칭 비용 발생
        - 생성에 제한이 없다.
            - 고객 요청이 너무 많이 오면 서버가 죽을 지도..(CPU, 메모리 임계..)

### 쓰레드 풀

![Image](https://github.com/user-attachments/assets/72b17a2b-5708-421c-8273-23a98bf00e1d)

- 수영장에 쓰레드가 놀고 있다가…(??) 요청이 오면 일을 한다.

200개 이후에는 쓰레드 대기, 거절

- 요청 마다 쓰레드 생성의 단점 보완
- 특징
    - 필요한 쓰레드를 쓰레드 풀에 보관하고 관리
    - 풀에 생성 가능한 쓰레드의 최대시를 관리. (톰캣은 최대 200개 기본 설정(변경 가능))
- 장점
    - 쓰레드가 미리 생성되어 있으니, 생성-종료 비용(CPU)가 절약, 응답 시간이 빠르다.
    - 생성 가능한 쓰레드의 최대치가 있으니, 과도한 요청이 들어와도 기존 요청을 안전하게 처리 가능

**팁**

- WAS의 주요 튜닝 포인트는 max thread 수
- 너무 낮게 설정?
    - 동시 요청이 많으면, 서버 리소스는 여유, 클라이언트는 응답 지연
- 너무 높게 설정?
    - 동시 요청이 많을 때, CPU, 메모리 리소스 임계점 초과로 서버 다운
- 장애 발생?
    - 클라우드 - 서버 늘리고, 이후 튜닝
    - 클라우드가 아니면 - 열심히 튜닝
- 적정 숫자?
    - 애플리케이션 로직의 복잡도, CPU, 메모리, IO 리소스 상황 고려해야함.. ← 잘하는 사람도 대략적으로 때려맞춤.
    - 성능 테스트 ← 트레픽이 좀 있는 서비스는 무조건 해야함.
        - 실제 서비스와 아주 유사하게 성능 테스트 시도
        - 툴 : ~~아파치 ab, 제이미터~~, nGrinder(좋음)
        - 트레픽을 받을 수 있구나~ 어? 왜이러지? 병목을 찾거나 할 수 있다.

### WAS의 멀티 쓰레드 지원

- was가 처리 해준다. 개발자가 멀티 쓰레드 관련 코드는 신경쓰지 않아도 됨.
- 싱글 쓰레드 프로그래밍 하듯이 편리하게 소스 코드 개발
    - 단, 멀티 쓰레드 환경 이므로 싱글톤 객체(서블릿, 스프링 빈)은 주의해서 사용!!!

## HTML, HTTP API, CSR, SSR

### 정적 리소스

- 주로 웹 브라우저

### HTML 페이지

동적 HTML 파일 생성 - jsp, thymeleaf (view templete)

- was가 동적인 html을 웹브라우저에 보여줌

### HTTP API

- HTML을 전달하는게 아니라, **데이터**를 전달
- 다양한 시스템에서 호출
    - 데이터만 주고 받음(주로 JSON 형식)
    - UI는 클라이언트가 별도 처리
        - 앱(아이폰, 안드로이드, PC 앱), 웹 클라이언트(자바 스크립트)
    - 서버 to 서버
        - 주문 서버 → 결제 서버
        - 기업간 데이터 통신

**백엔드 개발자 고민 포인트 3가지**

- 정적 리소스 어떻게 제공?
- 동적 html 페이지 어떻게 제공?
- HTTP API 어떻게 제공?

### SSR - 서버 사이드 렌더링

- 서버에서 최종 HTML을 생성해서 클라이언트에 전달
- 정적인 화면
- 관련 기술: JSP, 타임리프 → **백엔드 개발자**

### CSR - 클라이언트 사이드 렌더링

- HTML 결과를 자바스크립트를 사용해서 웹 브라우저에서 동적으로 생성
- 웹 환경을 마치 앱 처럼 필요한 부분부분을 변경할 수 있음
- 예) 구글지도, Gmail, 구글 캘린더
- 관련 기술: React, Vue.js → **프론트엔드 개발자**

 

- React, Vue.js를 CSR + SSR 동시에 지원하는 웹 프레임워크도 있음
- SSR을 사용하더라도, 자바스크립트를 사용해서 화면 일부를 동적으로 변경 가능

### 백엔드 개발자의 UI 기술

- JSP는 사장되었고.. 타임리프를 사용
- 하루면 다 알 수 있다 → 서버 사이드 렌더링 기술 학습 **필수**
- 웹 프론트엔드도 깊이있게 잘하려면 숙련, 오랜 시간이 필요
- 백엔드는 서버, DB, 인프라 등등 수 많은 백엔드 기술을 공부해야함.

## 자바 백엔드 웹 기술의 역사

- 서블릿 - 1997
    - HTML 생성이 어려움
- JSP - 1999
    - HTML 생성은 편리하지만, 비즈니스 로직까지 너무 많은 역할 담당
- 서블릿, JSP 조합 MVC 패턴 사용
    - 모델, 뷰 컨트롤러로 역할을 나누어 개발
- MVC 프레임워크 춘추 전국 시대 - 2000년 초 ~ 2010
    - MVC 패턴 자동화, 복잡한 웹 기술을 편리하게 사용할 수 있는 다양한 기능 지원
    - 스트럿츠, 웹워크, 스프링 MVC(과거 버전)
- **애노테이션 기반의 스프링 MVC** 등장 - 현재
    - @Controller
    - 단점이.. 없다…
- **스프링 부트**의 등장
    - 서버를 내장
    - 과거에 서버에 WAS 직접 설치, 소스는 War 파일 만들어서 설치한 WAS에 배포
    - 스프링 부트는 빌드 결과 (Jar)에 WAS 서버 포함 → 빌드 배포 단순화

스프링 웹 기술의 분화

- Web Servlet - Spring MVC (일반적으로 사용)
- Web Reactive - Spring WebFlux (최신 기술..인데 GPT가 그래도 아직 MVC를 많이 쓴다고 한다.)
    - 최소 쓰레드로 최대 성능
    - 함수형 스타일 - 동시처리 코드 효율화
    - 서블릿 기술 x
    - 기술적 난이도가 높음..
    - 아직 RDB 지원 부족 - 지금도 그런가?
    - 일반 MVC 쓰레드 모델도 충분히 빠름

### 뷰 템플릿의 역사

HTML을 편리하게 생성하는 뷰 기능

- JSP
    - 속도 느림, 기능 부족
- 프리마커, 벨로시티
    - 속도 문제 해결, 다양한 기능
- ⭐**타임리프(Thymeleaf)**
    - 내추럴 템플릿 : HTML의 모양을 유지
    - 스프링 MVC와 강력한 기능 통합
    - **최선의 선택**, 성능은 프리마커, 벨로시티가 좋거든.
